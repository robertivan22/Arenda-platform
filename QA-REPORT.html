🔴 Critical C-2 D112 API — req.json() not wrapped in try/catch + year/month not validated as numbers
Location apps/web/src/app/api/d112/route.ts — lines 30–40
Problem 1 const body = await req.json() throws an unhandled exception if the request body is not valid JSON (e.g. wrong Content-Type, empty body, network truncation). No try/catch means this surfaces as an unhandled 500.
Problem 2 as { year: number; month: number } is a compile-time cast only — it performs zero runtime validation. If a caller sends { "year": "abc", "month": "foo" }: !year is false (truthy string), month < 1 is false (NaN comparison). Validation passes. String(month).padStart(2, '0') then emits "foo" into a DB date string, producing a malformed periodTo value and corrupted report data.
Risk Unhandled 500 on invalid body; malformed date strings written into D112 report output; OWASP A03 (Injection — data).
api/d112/route.ts — fixTypeScript diff
  const body = await req.json() as { year: number; month: number }
  const { year, month } = body
  if (!year || !month || month < 1 || month > 12) {
    return NextResponse.json({ error: 'Parametrii lipsa: year, month (1-12)' }, { status: 400 })
  }

  let rawBody: unknown
  try { rawBody = await req.json() } catch {
    return NextResponse.json({ error: 'Body JSON invalid' }, { status: 400 })
  }
  const year  = Number((rawBody as any)?.year)
  const month = Number((rawBody as any)?.month)
  if (!Number.isInteger(year) || year < 2000 || year > 2100 ||
      !Number.isInteger(month) || month < 1 || month > 12) {
    return NextResponse.json({ error: 'Parametrii invalizi: year (2000-2100), month (1-12)' }, { status: 400 })
  }

🟠 High H-1 No .limit() on any list query — unbounded full-table scans on every page load
Location contracte/page.tsx line 26 · arendatori/page.tsx line 30 · plati/page.tsx lines 42, 87, 92 · all other list pages
Problem All queries fetch every row the authenticated user owns, on every component mount. With 500 lessors or 2 000 payments the response body becomes large, the browser holds the entire dataset in memory, and the initial page paint is blocked waiting for the full response. Client-side filtering (rows.filter(r => ...)) is applied after the full fetch — there is no server-side search or pagination.
Risk Performance degradation at scale; potential OOM crash in browser; poor UX for power users.

    .select('id, contract_number, ...')
    .order('created_at', { ascending: false })
    .limit(500)   // safety cap — replace with pagination long-term
    .then(({ data }) => { ... })

-- 4. MAKE YOURSELF ADMIN
-- ============================================================
UPDATE public.profiles
SET is_admin = TRUE
WHERE email = 'robert.ivan@lseg.com';

-- ============================================================
-- 4b. ADMIN RECOVERY HELPER
-- Run this in the Supabase SQL editor if admin access is lost:
--   SELECT grant_admin('your-email@example.com');
-- ============================================================
CREATE OR REPLACE FUNCTION public.grant_admin(target_email TEXT)
RETURNS VOID LANGUAGE plpgsql SECURITY DEFINER SET search_path = public AS $$
BEGIN
  UPDATE public.profiles SET is_admin = TRUE WHERE email = target_email;
  IF NOT FOUND THEN
    RAISE EXCEPTION 'No profile found for email: %', target_email;
  END IF;
END;
$$;
-- Revoke public execute — this function must be called from the SQL editor only
REVOKE EXECUTE ON FUNCTION public.grant_admin(TEXT) FROM PUBLIC, anon, authenticated;
