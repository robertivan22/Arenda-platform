export async function POST(req: NextRequest) {
  const authHeader = req.headers.get('Authorization')
  if (!authHeader?.startsWith('Bearer ')) {
    return NextResponse.json({ error: 'Neautorizat' }, { status: 401 })
  }

Part 2 — Scalability & Performance Audit

🔥 #1 Scalability Bottleneck — Leaflet map loads ALL parcel geometries with no pagination

MapParcelSelector.tsx issues two unbounded queries on load: parcele_fitosanitar.select('*') (all geometries with full GeoJSON blobs) and parcels.select(…).eq('status','ACTIVE') (all active registry parcels). Every row is then added as a L.GeoJSON Leaflet layer. A user with 200 parcels loads ~8–15 MB of polygon JSON and creates 200 DOM canvas paths simultaneously. The page becomes visually frozen at ~150 parcels on desktop and ~60 parcels on mobile. This is the first wall to hit in production, appearing around 50–80 subscribers who are power users with multi-county operations (400–500 parcels each).

PERF-01 — Harta Parcele loads ALL parcel geometries (unbounded, no clustering)
MapParcelSelector.tsx · lines 501, 1063–1079

What's wrong: Two queries have no .limit():

// Line 501 — all active registry parcels (no limit)
db.from('parcels').select('id,bloc_fizic,locality,surface').eq('status','ACTIVE').order('bloc_fizic')

// Line 1063 — all fitosanitar geometries (no limit, full geometry_geojson blob)

At 200 parcels: ~15 MB transferred. All rendered as Leaflet polygon layers. The browser freezes on mobile. No viewport culling, no clustering, no simplification.

Fix (short-term): Add .limit(300) on both queries.
Fix (proper): Switch to bounds-based loading — query only parcels whose lat/lng centroid falls within the current map bounds. Re-query on moveend with debounce. Add Leaflet cluster layer (leaflet.markercluster) for the registry pin layer. Simplify polygon geometry to 4 decimal places (~11m precision) before storing.

PERF-03 — /api/farm-dashboard: N×2 uncapped parallel external API calls
lib/farm-dashboard/index.ts · processParcel()

What's wrong: The orchestrator calls Promise.allSettled(inputs.map(processParcel)). Each parcel fires 2 external requests (Open-Meteo + Sentinel Hub) simultaneously. At 50 parcels (the capped limit), that's 100 concurrent outbound HTTP calls from a single Cloudflare Worker invocation. Sentinel Hub rate-limits OAuth clients at ~10 req/s; 50 parcels will hit the limit and return partial 429 errors silently swallowed by allSettled.

Fix: Add a concurrency gate — process parcels in batches of 5–8 instead of fully parallel: 
async function processInBatches(inputs: ParcelInput[], batchSize = 6) {
  const results: ParcelResult[] = []
  for (let i = 0; i < inputs.length; i += batchSize) {
    const batch = inputs.slice(i, i + batchSize)
    results.push(...await Promise.all(batch.map(processParcel)))
  }
  return results
}

PERF-07 — Sentinel Hub access token cached in module-level variable (cold-start loss)
lib/farm-dashboard/sentinel-hub.ts · line ~70

What's wrong: let cachedToken: … is a module-level variable. Cloudflare Workers isolates are recycled after ~30s of inactivity, discarding the cached token. Under bursty usage this means 1–2 extra OAuth token fetches per request wave (100–200ms extra latency).

MapParcelSelector.tsx — all parcel GeoJSON rendering

INFRA-01 — Cloudflare Pages function timeout risk for farm-dashboard with 50 parcels
api/farm-dashboard/route.ts & Cloudflare Pages limits

What's wrong: Cloudflare Pages Functions have a 30-second CPU time limit (50ms CPU per request on Free, 30s on Paid). With 50 parcels × 2 external requests (Open-Meteo + Sentinel Hub) running in parallel, the total wall-clock time can reach 8–15 seconds on slow government APIs. At high load, a Sentinel Hub OAuth token fetch (cold start) adds 500–800ms. If any external call stalls for 30+ seconds, the entire request times out with no partial result.

Fix: Add per-request timeouts using AbortSignal.timeout(8000) on each external fetch. Return partial results for parcels that timed out with no_data: true instead of blocking the entire response.
