N-H2 api/apia/route.ts — unguarded req.json() crashes edge function on malformed body HIGH

The APIA export API endpoint calls req.json() without a try/catch. If the caller sends a malformed (non-JSON) body, the edge function throws an unhandled exception and returns a 500 error instead of a clean 400 validation response. This is the same issue as old C-2 (fixed in api/d112/route.ts) but was missed for this endpoint.

Current Code — apps/web/src/app/api/apia/route.ts
const body = await req.json() as { campaignYear: number }  // ← throws if body is not valid JSON
const { campaignYear } = body
if (!campaignYear) {
  return NextResponse.json({ error: 'Parametru lipsă: campaignYear' }, { status: 400 })
}
Fix (mirror d112/route.ts pattern)
let rawBody: unknown
try { rawBody = await req.json() } catch {
  return NextResponse.json({ error: 'Body JSON invalid' }, { status: 400 })
}
const campaignYear = Number((rawBody as any)?.campaignYear)
if (!Number.isInteger(campaignYear) || campaignYear < 2000 || campaignYear > 2100) {
  return NextResponse.json({ error: 'Parametru invalid: campaignYear (2000-2100)' }, { status: 400 })
}

N-M5 fitosanitar/page.tsx — user search string interpolated into PostgREST filter expression MEDIUM

The full-text search on the fitosanitar registry page interpolates the user's raw search input directly into a PostgREST .or() filter string. PostgREST treats this as a query expression, so special characters like ., (, ), or commas can break the filter syntax or cause unexpected filter behaviour. While this is not a traditional SQL injection (PostgREST parameterises values), it does allow filter injection that can bypass intended query constraints.

Current Code — apps/web/src/app/(app)/fitosanitar/page.tsx
if (search.trim()) {
  q = q.or(
    `denumire_produs.ilike.%${search.trim()}%,agent_daunare.ilike.%${search.trim()}%,locul_terenului.ilike.%${search.trim()}%`
    // ← user input directly in filter expression — commas/dots could break or inject
  )
}
Fix — sanitize before interpolating into PostgREST filter
if (search.trim()) {
  // Escape % and _ for ILIKE; strip PostgREST metacharacters (, . ( ) )
  const safe = search.trim()
    .replace(/[%_\\]/g, '\\$&')   // escape ILIKE wildcards
    .replace(/[,()`'"]/g, '')      // remove PostgREST expression metacharacters
  q = q.or(
    `denumire_produs.ilike.%${safe}%,agent_daunare.ilike.%${safe}%,locul_terenului.ilike.%${safe}%`
  )
}

N-L1 auth/callback/route.ts — unvalidated 'next' parameter used in redirect LOW

The OAuth/magic-link callback handler reads the next query parameter and uses it directly in a redirect. If an attacker crafts a link like /auth/callback?code=valid&next=//evil.com, the response would be NextResponse.redirect('https://arenda.pro//evil.com'). Modern browsers will typically resolve //evil.com as a protocol-relative URL before the host concatenation takes effect, making exploitation difficult — but it is still best practice to validate the parameter.

Current Code — apps/web/src/app/auth/callback/route.ts
const next = searchParams.get('next') ?? '/dashboard'  // ← unvalidated
// ...
return NextResponse.redirect(`${origin}${next}`)        // ← open redirect risk
Fix
const rawNext = searchParams.get('next') ?? '/dashboard'
// Ensure next is a valid relative path (starts with / but not //)
const next = rawNext.startsWith('/') && !rawNext.startsWith('//') ? rawNext : '/dashboard'
return NextResponse.redirect(`${origin}${next}`)
