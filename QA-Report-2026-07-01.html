Every row in the D112 dataset is emitted with paymentType: 'CASH' regardless of the actual payment_type value stored in the transactions table. ● MEDIUM P-01 · leaflet/dist/leaflet.css imported at top-level of two components — becomes globally applied after first map visit, potentially affecting non-map page styles
apps/web/src/components/MapParcelSelector.tsx · line 5 · import 'leaflet/dist/leaflet.css'

Both map components import Leaflet's CSS at module level. Because they are loaded via dynamic({ ssr: false }), the CSS is not injected at SSR time. However, once the map page is first visited in a session, Next.js injects the Leaflet CSS globally into <head>. Leaflet's CSS includes generic selectors like .leaflet-container img, path, and .leaflet-popup-content-wrapper that can unexpectedly override app styles on other pages visited after the map in the same session.

// Or import CSS conditionally inside useEffect:

  import('leaflet/dist/leaflet.css')

apps/web/src/components/MapParcelSelector.tsx · line 62 · minZoom?: number interface · line 75 · minZoom: 12 in APIA layer config

● LOW P-03 · assistant/route.ts Supabase queries use hardcoded .limit(50/60/100) — no per-user data volume management
apps/web/src/app/api/assistant/route.ts · fetchLiveData() · lines 73–81
The AI assistant context builder fetches up to 50 contracts, 60 stock items, 100 transactions etc. and serializes them all into a Groq prompt. As a farm grows, prompt tokens will approach and exceed Groq's context window limit (128k tokens for llama-3.3-70b), at which point the call silently truncates or errors. There is no token budget check before submission.
Action
Add a token estimate before calling Groq (e.g., JSON.stringify(promptData).length / 4 as a rough token count). If it exceeds ~100k, trim lower-priority data arrays (stock > transactions) and log a warning.

● HIGH SEC-01 · Storage RLS policy documents_auth checks only auth.uid() IS NOT NULL — any authenticated user can access any other user's contract documents
supabase-fix-rls-defaults.sql · lines ~48–52 · documents_auth policy
The documents storage bucket policy is: USING (bucket_id = 'documents' AND auth.uid() IS NOT NULL). This allows any authenticated user to read, update, or delete any file in the bucket as long as they are logged in — not just their own files. Contract PDFs, identity documents, and payment receipts from all tenants are cross-accessible. This is a multi-tenant data isolation failure.
Fix — scope to user's own folder
-- Drop the overly-permissive policy:
DROP POLICY IF EXISTS "documents_auth" ON storage.objects;

-- Replace with path-scoped policy (files stored as {user_id}/...):
CREATE POLICY "documents_own" ON storage.objects
  FOR ALL
  USING (
    bucket_id = 'documents'
    AND (storage.foldername(name))[1] = auth.uid()::text
  )
  WITH CHECK (
    bucket_id = 'documents'
    AND (storage.foldername(name))[1] = auth.uid()::text
  );

-- Ensure upload paths use user_id prefix:
-- e.g., storage_path = `${user.id}/contracts/${filename}`
● HIGH SEC-02 · Shapefile and contract document file-type validation is client-side only — server accepts any MIME type
apps/web/src/components/distribuire/ImportWizardModal.tsx · line 168 · client-side accept filter
ImportWizardModal uses accept=".zip,.geojson,.json" and client-side MIME checks. The Supabase storage bucket allowed_mime_types for the documents bucket is set to PDF/images only, which partially protects contract attachments. However, the shapefile import appears to go through a direct client upload path where a crafted request bypassing the browser can upload arbitrary files. Any server-side API route that processes uploaded files should validate MIME type and file magic bytes server-side.
Fix — validate server-side in the import handler
// In the import route/server action:
const ALLOWED_TYPES = new Set(['application/zip', 'application/x-zip-compressed', 'application/geo+json', 'application/json'])
const MAX_SIZE = 10 * 1024 * 1024 // 10 MB

if (!ALLOWED_TYPES.has(file.type) || file.size > MAX_SIZE) {
  return NextResponse.json({ error: 'Tip sau dimensiune fisier neacceptata' }, { status: 400 })
}

● MEDIUM SEC-04 · efactura/submit stores raw XML (including full invoice data) in efactura_submissions.xml_sent with no RLS verified on that table
apps/web/src/app/api/efactura/submit/route.ts · line ~130 · insert into efactura_submissions
efactura_submissions stores the complete UBL 2.1 XML including CIF, IBAN, transaction amounts and personal data. This table is not referenced in supabase-rls.sql or any migration file, meaning it may have no RLS policy. If RLS is disabled on this table, any authenticated user can query all other users' submitted invoice XMLs. Verify and add the policy.

● LOW SEC-05 · /api/efactura/status does not validate that invoice_id is a UUID — injectable into downstream URL construction
apps/web/src/app/api/efactura/status/route.ts · ~line 22 · invoice_id from query param
invoice_id is taken from the URL query string and passed to db.from('invoices').eq('id', invoice_id). PostgREST parameterizes this correctly so SQL injection is not possible, but if invoice_id is also used in log strings or downstream ANAF API URL construction without sanitization, it's worth guarding. Add a UUID format check for defense-in-depth.
Fix — one-line guard
const UUID_RE = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i
if (!invoice_id || !UUID_RE.test(invoice_id)) {
  return NextResponse.json({ error: 'invoice_id invalid' }, { status: 400 })
}

● MEDIUM A-01 · Bell and Profile interactive areas are <div> elements — not keyboard-reachable without explicit tabIndex and ARIA role
apps/web/src/components/layout/AppTopbar.tsx · bell wrapper div · profile wrapper div
Both the bell notification trigger and the profile button in the topbar are <div> elements with onClick and cursor-pointer. ● LOW A-02 · Several form <input> and <select> elements use only placeholder text without an associated <label>
Multiple page.tsx files — parcele, contracte, fitosanitar, setari — unlabeled inputs
Multiple inline inputs were found where the label text exists visually (via adjacent <label>) but the label lacks a matching htmlFor / id pairing, meaning screen readers cannot associate them. Some inputs rely solely on placeholder text which disappears once the user types and does not constitute an accessible label.
Fix pattern
// Before (bad):
<label>Suprafata (ha)</label>
<input className={inputCls} type="number" />

// After (correct):
<label htmlFor="surface-input">Suprafata (ha)</label>
<input id="surface-input" className={inputCls} type="number" />
