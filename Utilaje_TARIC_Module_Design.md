# Utilaje TARIC Module — Design & Implementation Notes

## Purpose

This document describes the recommended design for a **Utilaje TARIC module** that validates agricultural machinery commodity codes using a **local TARIC mirror**.

The module focuses only on:

- TARIC / NC / HS commodity-code validation
- local TARIC dataset ingestion
- agricultural machinery candidate-code mapping
- internal validation API

It does **not** include RO e-Transport or UIT logic.

---

## Conceptual distinction

### TARIC / NC / HS code

A TARIC, NC, or HS code is a **commodity classification code**.

Typical levels are:

- **HS code** — 6 digits, international level
- **CN / NC code** — 8 digits, EU Combined Nomenclature level
- **TARIC code** — usually 10 digits, EU import-measure level

These codes can be validated against TARIC for:

- format
- existence
- validity date
- description
- applicable customs measures
- additional codes
- certificates or document requirements
- geographical origin or destination measures

### UIT code

A UIT code is different. It is not a commodity code and should not be generated by this module.

UIT is transport/reporting logic and should remain in a separate RO e-Transport module if needed later.

## Recommended architecture

```text
Official TARIC full / daily data
        ↓
Ingestion job
        ↓
Raw XML / archive storage
        ↓
Parser / normalizer
        ↓
Relational TARIC database
        ↓
Validation API
        ↓
Utilaje classification algorithm
```

The core idea is to avoid scraping the TARIC UI and instead maintain a local, auditable TARIC dataset.

---

## Integration strategy

### Initial load

1. Download the official TARIC full database file or a trusted member-state TARIC distribution file.
2. Store the raw archive unchanged.
3. Compute and store a SHA-256 hash.
4. Parse XML files into staging tables.
5. Validate basic row counts and required entities.
6. Promote staging data into production tables.

### Daily update

1. Download the daily update or diff file.
2. Store the raw update archive unchanged.
3. Compute and store a SHA-256 hash.
4. Parse update XML.
5. Apply changes transactionally.
6. Preserve historical validity periods.
7. Log the snapshot used for every validation result.

TARIC ingestion logic.

- load additional codes
- load certificate codes
- load geographical areas

Important production note:

---

### `taric_validation.py`

Validation engine.

Responsibilities:

- normalize codes
- validate code format
- detect code level: HS, CN, TARIC
- check whether the code exists in the local TARIC dataset
- check whether the code is valid for a reference date
- retrieve related measures
- return structured validation response

Validation levels:

```text
6 digits  -> HS
8 digits  -> CN / NC
10 digits -> TARIC
```

---

### `utilaje_classifier.py`

Agricultural machinery classification logic.

Responsibilities:

- inspect product description and attributes
- generate candidate TARIC codes
- assign confidence score
- mark ambiguous cases for manual review
- call TARIC validation for the recommended candidate

## Recommended database model

### `taric_snapshot`

Stores each loaded TARIC file or update package.

```sql
create table taric_snapshot (
    snapshot_id    bigserial primary key,
    source_name    text not null,
    source_file    text not null,
    source_hash    text not null unique,
    loaded_at      timestamp not null default now(),
    reference_date date,
    is_active      boolean not null default true
);
```

---

### `goods_nomenclature`

Stores valid goods codes and descriptions.

```sql
create table goods_nomenclature (
    id           bigserial primary key,
    goods_code   varchar(10) not null,
    suffix       varchar(4),
    language     varchar(2) not null,
    description  text,
    valid_from   date,
    valid_to     date,
    product_line varchar(10),
    snapshot_id  bigint not null
);
```

---

### `taric_measure`

Stores measures linked to commodity codes.

```sql
create table taric_measure (
    id                bigserial primary key,
    measure_sid       varchar(50),
    goods_code        varchar(10) not null,
    geographical_area varchar(10),
    measure_type      varchar(20),
    additional_code   varchar(20),
    duty_expression   text,
    regulation_id     varchar(50),
    valid_from        date,
    valid_to          date,
    snapshot_id       bigint not null
);
```

## Core validation flow

```text
Input:
  code
  reference_date
  language
  origin_country optional

Step 1: Normalize code
Step 2: Validate format
Step 3: Determine level: HS / CN / TARIC
Step 4: Search goods_nomenclature for code valid on reference date
Step 5: Retrieve description
Step 6: Retrieve measures for code and origin country if available
Step 7: Return structured response
```

## Utilaje classification logic

The module should use deterministic rules first, not AI.

Recommended approach:

```text
Product description + attributes
        ↓
Candidate-code rules
        ↓
Confidence score
        ↓
TARIC validation
        ↓
Recommended code or manual review
```

## Manual-review rules

The classifier should return `requires_manual_review = true` when:

- the description is too generic
- the product could fit multiple headings
- the item is a part and the parent machine is unclear
- the machine has a special function not covered by the simple mapping
- the code exists only at HS/CN level and needs a more specific TARIC split
- TARIC validation fails for the reference date

## Production recommendations

1. Use official TARIC data as the legal reference.
2. Do not scrape the TARIC web interface.
3. Prefer full database + daily update files.
4. Store raw source files unchanged.
5. Store SHA-256 hashes for auditability.
6. Keep validity periods instead of overwriting old data.
7. Log the `snapshot_id` used for each validation result.
8. Use PostgreSQL indexes on code, language, date range, and geographical area.
9. Separate classification logic from validation logic.
10. Keep RO e-Transport / UIT outside this module.

## Summary

The recommended implementation is a deterministic **rules + validation engine**:

```text
Utilaj attributes
        ↓
Candidate TARIC code
        ↓
Local TARIC validation
        ↓
Recommended code or manual review
```

This gives the Utilaje module a stable and auditable foundation while keeping transport-specific RO e-Transport logic out of scope.
